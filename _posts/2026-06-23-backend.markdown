## The Problem: The Missing Backbone for Multi‑User Vision

Most computer vision projects focus on the model — the AI that detects objects, anomalies, or events. But in the real world, a vision system is useless without a robust backend to:

- **Manage multiple users** with different roles and permissions
- **Ingest frames from many cameras** simultaneously
- **Stream frames in real time** to authorized clients
- **Track events** (detections, alerts) per camera
- **Scale** from a single camera to hundreds

Existing solutions are often monolithic, tightly coupled to specific hardware, or lack proper access control. I set out to build a **pluggable, extensible backend** that separates concerns cleanly — making it easy to swap out components, add new features, or scale specific parts of the system.

## The Solution: A Modular, Clean‑Architecture Backend

> *"I treated this like a production microservice — even though it's a prototype, the architecture is designed to scale."*

### Core Design Principles

**1. Clear Separation of Concerns**

The system is structured into distinct layers:

```
API Layer (FastAPI)
    ↓
Business Logic (Services)
    ↓
Data Storage (SQLModel + Redis / In-Memory)
    ↓
Repositories (Abstracted Frame Storage)
```

This makes it easy to extend, replace components, or scale specific parts independently.

**2. Dual Authentication**

| Actor | Authentication Method |
|-------|----------------------|
| **Users** | JWT (JSON Web Tokens) — standard, stateless, and widely supported |
| **Cameras** | API keys — generated securely, hashed with bcrypt, and never stored in plaintext |

**3. Real‑Time Streaming via WebSockets**

Clients can subscribe to live frame updates from any camera they have access to, enabling near‑instantaneous visual feedback.

**4. Pluggable Frame Storage**

Frames can be stored in **Redis** (for fast, ephemeral access) or **in‑memory** (for development). The repository pattern abstracts the storage layer, so swapping between them is trivial.

**5. Role‑Based Access Control**

Each camera has an **owner** (who can manage it) and multiple **viewers** (who can only view frames).

## Key Lessons Learned

This project reinforced that **backend architecture is just as important as the vision pipeline** — and that a well‑designed prototype is the fastest path to production.

1. **Separation of concerns pays off:** By isolating the API, business logic, and data layers, I made the system trivially extensible. Adding a new feature (like event tracking) didn't require rewriting core components.

2. **Authentication is non‑negotiable:** Even in a prototype, I implemented proper JWT and bcrypt‑hashed API keys. This forced me to think about security from day one — and made the system immediately usable in real scenarios.

3. **Pluggable storage is a superpower:** The repository pattern means I can swap Redis for a cloud storage solution (S3, Azure Blob) without changing a single line of business logic.

4. **WebSockets are the right tool for real‑time:** HTTP polling would never scale for live frame delivery. WebSockets provide low‑latency, bi‑directional communication that feels instantaneous to the user.

5. **Documentation is architecture:** The code is heavily commented, and the README clearly lists limitations and future plans. This makes the project approachable for collaborators and demonstrates engineering maturity.
